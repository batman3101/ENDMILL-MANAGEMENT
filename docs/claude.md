# **Vibe 코딩 환경을 위한 SOLID 원칙 가이드라인**

## **단일 책임 원칙 (Single Responsibility Principle \- SRP)**

설명:  
하나의 클래스나 모듈은 오직 하나의 변경 이유만을 가져야 합니다. 즉, 하나의 액터(actor)에 대한 책임만 수행해야 합니다.  
Vive 코딩 Rule:  
\[SRP\] Rule: 클래스는 하나의 기능만 담당하고, 변경될 이유도 하나여야 합니다.  
**예시:**

**// Bad Example (규칙 위반)**

\# 사용자 관리와 이메일 전송이라는 두 가지 책임을 가짐  
class UserManager:  
    def create\_user(self, name):  
        \# 사용자 생성 로직  
        print(f"User {name} created.")  
      
    def send\_welcome\_email(self, email):  
        \# 이메일 전송 로직  
        print(f"Welcome email sent to {email}.")

**// Good Example (규칙 준수)**

\# 사용자 생성 책임만 가짐  
class UserCreator:  
    def create\_user(self, name):  
        \# 사용자 생성 로직  
        print(f"User {name} created.")

\# 이메일 전송 책임만 가짐  
class EmailSender:  
    def send\_email(self, email\_address, subject, body):  
        \# 이메일 전송 로직  
        print(f"Email sent to {email\_address} with subject '{subject}'.")

## **개방-폐쇄 원칙 (Open-Closed Principle \- OCP)**

설명:  
소프트웨어 개체(클래스, 모듈, 함수 등)는 확장에 대해서는 열려 있어야 하지만, 수정에 대해서는 닫혀 있어야 합니다. 기존 코드를 변경하지 않고도 기능을 추가할 수 있어야 합니다.  
Vive 코딩 Rule:  
\[OCP\] Rule: 기능 확장은 허용하되, 기존 코드 수정은 피해야 합니다. 상속이나 인터페이스를 활용하세요.  
**예시:**

**// Bad Example (규칙 위반)**

\# 새로운 결제 방법이 추가될 때마다 process\_payment 메서드를 수정해야 함  
class PaymentProcessor:  
    def process\_payment(self, amount, method):  
        if method \== "credit\_card":  
            print(f"Processing ${amount} with credit card.")  
        elif method \== "paypal":  
            print(f"Processing ${amount} with PayPal.")  
        \# 새로운 결제 방식(예: "crypto")이 추가되면 이 코드를 수정해야 함

**// Good Example (규칙 준수)**

from abc import ABC, abstractmethod

\# 추상 결제 방법 인터페이스  
class PaymentMethod(ABC):  
    @abstractmethod  
    def pay(self, amount):  
        pass

\# 구체적인 결제 방법 클래스들  
class CreditCardPayment(PaymentMethod):  
    def pay(self, amount):  
        print(f"Processing ${amount} with credit card.")

class PayPalPayment(PaymentMethod):  
    def pay(self, amount):  
        print(f"Processing ${amount} with PayPal.")

\# 새로운 결제 방법 추가 시 기존 코드를 수정할 필요 없음  
class CryptoPayment(PaymentMethod):  
    def pay(self, amount):  
        print(f"Processing ${amount} with Crypto.")

\# 결제 처리기는 어떤 결제 방법이든 처리 가능  
class PaymentProcessor:  
    def process\_payment(self, payment\_method: PaymentMethod, amount):  
        payment\_method.pay(amount)

## **리스코프 치환 원칙 (Liskov Substitution Principle \- LSP)**

설명:  
하위 타입은 언제나 상위 타입으로 교체할 수 있어야 합니다. 즉, 하위 클래스는 상위 클래스의 계약(메서드의 동작 방식, 파라미터 등)을 위반하지 않고 그 역할을 수행해야 합니다.  
Vive 코딩 Rule:  
\[LSP\] Rule: 하위 클래스는 상위 클래스의 역할을 완벽히 대체할 수 있어야 합니다.  
**예시:**

**// Bad Example (규칙 위반)**

\# Square가 Rectangle을 상속하지만, set\_width/set\_height의 동작이 부모와 다름  
class Rectangle:  
    def \_\_init\_\_(self, width, height):  
        self.\_width \= width  
        self.\_height \= height

    def set\_width(self, width):  
        self.\_width \= width

    def set\_height(self, height):  
        self.\_height \= height

    def get\_area(self):  
        return self.\_width \* self.\_height

class Square(Rectangle):  
    def \_\_init\_\_(self, size):  
        super().\_\_init\_\_(size, size)

    def set\_width(self, width):  
        self.\_width \= width  
        self.\_height \= width

    def set\_height(self, height):  
        self.\_width \= height  
        self.\_height \= height

\# 이 함수는 Rectangle 타입에 대해 정상 동작을 기대함  
def print\_area(rect: Rectangle):  
    rect.set\_width(5)  
    rect.set\_height(4)  
    \# 기대값은 20이지만, Square 객체가 들어오면 16이 출력되어 LSP를 위반함  
    print(f"Area: {rect.get\_area()}")

**// Good Example (규칙 준수)**

from abc import ABC, abstractmethod

\# 공통 인터페이스(도형) 정의  
class Shape(ABC):  
    @abstractmethod  
    def get\_area(self):  
        pass

\# 각 도형은 자신만의 로직을 구현  
class Rectangle(Shape):  
    def \_\_init\_\_(self, width, height):  
        self.width \= width  
        self.height \= height

    def get\_area(self):  
        return self.width \* self.height

class Square(Shape):  
    def \_\_init\_\_(self, size):  
        self.size \= size

    def get\_area(self):  
        return self.size \*\* 2

\# 어떤 Shape 타입이든 문제없이 동작  
def print\_area(shape: Shape):  
    print(f"Area: {shape.get\_area()}")

## **인터페이스 분리 원칙 (Interface Segregation Principle \- ISP)**

설명:  
클라이언트는 자신이 사용하지 않는 인터페이스에 의존하도록 강요받아서는 안 됩니다. 하나의 거대한 인터페이스보다 여러 개의 구체적인 인터페이스가 낫습니다.  
Vive 코딩 Rule:  
\[ISP\] Rule: 클라이언트가 사용하지 않는 기능까지 포함하는 거대한 인터페이스 대신, 역할에 맞는 작은 인터페이스 여러 개를 만드세요.  
**예시:**

**// Bad Example (규칙 위반)**

from abc import ABC, abstractmethod

\# 너무 많은 기능을 가진 거대한 인터페이스  
class Machine(ABC):  
    @abstractmethod  
    def print\_document(self): pass  
      
    @abstractmethod  
    def scan\_document(self): pass

    @abstractmethod  
    def fax\_document(self): pass

\# 프린터는 스캔이나 팩스 기능이 없지만, 관련 메서드를 강제로 구현해야 함  
class SimplePrinter(Machine):  
    def print\_document(self):  
        print("Printing document.")  
      
    def scan\_document(self):  
        raise NotImplementedError("Scanner not supported")  
          
    def fax\_document(self):  
        raise NotImplementedError("Fax not supported")

**// Good Example (규칙 준수)**

from abc import ABC, abstractmethod

\# 기능별로 인터페이스를 분리  
class Printer(ABC):  
    @abstractmethod  
    def print\_document(self): pass

class Scanner(ABC):  
    @abstractmethod  
    def scan\_document(self): pass

\# 필요한 인터페이스만 구현  
class SimplePrinter(Printer):  
    def print\_document(self):  
        print("Printing document.")

class Photocopier(Printer, Scanner):  
    def print\_document(self):  
        print("Printing document.")  
      
    def scan\_document(self):  
        print("Scanning document.")

## **의존성 역전 원칙 (Dependency Inversion Principle \- DIP)**

설명:  
상위 수준 모듈은 하위 수준 모듈에 의존해서는 안 되며, 둘 모두 추상화에 의존해야 합니다. 추상화는 세부 사항에 의존해서는 안 되며, 세부 사항이 추상화에 의존해야 합니다.  
Vive 코딩 Rule:  
\[DIP\] Rule: 구체적인 구현 클래스에 직접 의존하지 말고, 추상화된 인터페이스나 상위 클래스에 의존하세요.  
**예시:**

**// Bad Example (규칙 위반)**

\# 상위 모듈(Notification)이 하위 모듈(EmailSender)에 직접 의존  
class EmailSender:  
    def send(self, message):  
        print(f"Sending email: {message}")

class Notification:  
    def \_\_init\_\_(self):  
        self.sender \= EmailSender() \# 구체 클래스에 직접 의존

    def send\_notification(self, message):  
        self.sender.send(message)

**// Good Example (규칙 준수)**

from abc import ABC, abstractmethod

\# 추상화된 인터페이스  
class MessageSender(ABC):  
    @abstractmethod  
    def send(self, message):  
        pass

\# 하위 모듈은 추상화에 의존하여 구현  
class EmailSender(MessageSender):  
    def send(self, message):  
        print(f"Sending email: {message}")

class SMSSender(MessageSender):  
    def send(self, message):  
        print(f"Sending SMS: {message}")

\# 상위 모듈은 추상화(MessageSender)에 의존 (의존성 주입 사용)  
class Notification:  
    def \_\_init\_\_(self, sender: MessageSender):  
        self.sender \= sender

    def send\_notification(self, message):  
        self.sender.send(message)

\# 사용 시점에 구체적인 구현을 주입  
email\_notifier \= Notification(EmailSender())  
sms\_notifier \= Notification(SMSSender())
